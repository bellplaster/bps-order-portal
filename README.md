# BPS Order Portal — Cloudflare Native

This version removes Google Apps Script entirely.

## Runtime architecture

BPS portal
→ Cloudflare Pages Functions
→ XLSX generation inside Cloudflare
→ R2 archive
→ D1 order and diagnostic history
→ Cloudflare Email Service REST API
→ Bell Plaster email

## Important design decision

The uploaded `reference/Accrivia_Skeleton.xlsx` is used as the source of truth
for the workbook package structure.

At runtime, the generator does not need Google Drive or Excel. The skeleton's
static XML package files are embedded in `functions/_shared/xlsx.js`, while
`sheet1.xml` is generated for each order.

This keeps the known Accrivia-compatible structure while removing the Google
dependency.

## Cloudflare resources required

Create:

1. D1 database
   - Suggested name: `bps-order-portal`
   - Pages binding name: `DB`

2. R2 bucket
   - Suggested name: `bps-order-files`
   - Pages binding name: `ORDER_FILES`

3. Cloudflare Email Sending
   - Verify a sender domain/address
   - Create an API token with email sending permission

## D1 setup

Run:

```text
migrations/0001_initial.sql
```

through the D1 dashboard console.

## Pages bindings

In the Pages project:

Settings
→ Bindings
→ Add

Add:

```text
DB          = the D1 database
ORDER_FILES = the R2 bucket
```

Redeploy after bindings are added.

## Encrypted variables and secrets

Keep:

```text
PORTAL_PASSWORD
SESSION_SECRET
```

Remove the old Google values:

```text
APPS_SCRIPT_URL
BRUNSWICK_WEBHOOK_KEY
```

Add:

```text
CLOUDFLARE_ACCOUNT_ID
CLOUDFLARE_EMAIL_API_TOKEN
EMAIL_FROM
EMAIL_TO
EMAIL_CC
```

`EMAIL_CC` may be blank.

## Acceptance sequence

Before live submission:

1. Upload both files generated by `npm run test:xlsx` into Accrivia.
2. Confirm the Cloudflare-native files import correctly.
3. Create and bind D1 and R2.
4. Configure Cloudflare Email Sending.
5. Deploy this package.
6. Submit a test order through the portal.
7. Confirm:
   - separate GF and L1 XLSX files are emailed;
   - files exist in R2;
   - D1 shows `completed`;
   - `/api/status` reports the latest stage and exact error if failure occurs.

## Diagnostics

Authenticated endpoint:

```text
/api/status
```

It returns:

- latest D1 order record;
- latest generated file records;
- stage history;
- exact error message and stack trace.

## R2 layout

```text
orders/YYYY/MM/SUBMISSION_ID/
  BPS-REFERENCE-GF.xlsx
  BPS-REFERENCE-L1.xlsx

```
